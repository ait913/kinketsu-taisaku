# 金欠対策 (kinketsu-taisaku) — MVP コア設計

> 対象: 確定/未確定レコード + カテゴリ⊃タグ + 定期レコード(materialize) + 残高推移折れ線 + 多月 forecast
> スタック: Vite8+React19+TS+Tailwind4+TanStack Query / Hono4+Drizzle / SQLite(better-sqlite3) / better-auth 1.6.14 (Magic Link + Google OAuth) / appily(Coolify)
> 前提: [`Muraki/projects/_pre/research-moneylog-successor-20260608.md`](../../_pre/research-moneylog-successor-20260608.md)、[`CLAUDE.md`](../CLAUDE.md)

## 目的

サブスク/クレカ/給料などの定期収支を「月次ルール」で登録し、未確定レコードを未来へ実体生成 (materialize) することで、**現在残高 (確定のみ)** と **月末着地残高 (未確定込み予測)** の 2 系列を多月先まで可視化する家計簿。ユーザーは Touri 本人 (永続アカウント、Magic Link / Google OAuth)。

moneylog からは**概念のみ**継承 (確定/未確定 + 着地予測 + カテゴリ⊃タグ + bundle)。実装慣習・単一エンドポイント API はコピーせず、Hono REST + materialize でモダンに再構築。

## 1. 用語と数理モデル (Reviewer 必読)

| 用語 | 定義 |
|---|---|
| record | 単一 shape の取引。`paid` boolean で確定/未確定を区別。未確定 = 予定 |
| paid=true | 確定。実際に動いたお金 |
| paid=false | 未確定 = 予定。予測にのみ効く |
| signedAmount | 符号付き整数 (円、小数なし)。収入=正 / 支出=負 |
| category | 分類の束ね。`id=1`収入 / `id=2`支出 をシード。それ以外は符号フリー |
| tag | category に属する最小分類。色を持つ (nullable) |
| anchor (初期残高アンカー) | 累積残高計算の起点。ユーザー設定の基準残高 + 基準日 |
| recurring rule | 月次ルール。dayOfMonth(1-31)+signedAmount+category/tag+description |
| materialize | rule から未確定 record を未来各月に実体生成すること |
| yearMonth | `"YYYY-MM"` 文字列 (例 `"2026-06"`) |

### 1.1 残高の数理 (唯一の定義。実装はこれに従う)

通貨は**整数の円**で保持・計算する (浮動小数を一切使わない)。

- **anchor**: `{ balance: int (円), asOf: "YYYY-MM-DD" }`。「asOf 時点でちょうど balance 円ある」を意味する起点。
- **現在残高 (currentBalance)** = `anchor.balance + Σ(signedAmount for record where paid=true AND date >= anchor.asOf AND date <= today)`
  - anchor.asOf より前の確定 record は計算に含めない (anchor が起点なので二重計上を避ける)。
- **ある月 M の月末着地残高 (endingBalanceForecast[M])**:
  ```
  monthDelta(M, includeUnpaid) =
      Σ signedAmount for record where date ∈ M
        AND (includeUnpaid ? true : paid=true)
  endingBalanceForecast[M] = startingBalance[M] + monthDelta(M, includeUnpaid=true)
  endingBalanceConfirmed[M] = startingBalance[M_confirmed] + monthDelta(M, includeUnpaid=false)
  ```
- **繰り越し (running balance)**: 月をまたぐ累積は明示的に計算する。
  ```
  anchorMonth = anchor.asOf の yearMonth
  startingBalance[anchorMonth] = anchor.balance
      + Σ signedAmount for record where date < anchor.asOf AND date ∈ anchorMonth AND paid=true
      // anchor.asOf が月途中の場合、その月頭〜asOf 前日の確定分を戻す必要があるが、
      // MVP は「anchor.asOf は月初(=1日)を推奨」とし、月途中 anchor の場合は
      // startingBalance[anchorMonth] = anchor.balance とし、anchor.asOf 当日含む同月確定分を monthDelta に含める。
  startingBalance[M]   = endingBalanceForecast[M-1]   // forecast 系列
  startingBalanceConfirmed[M] = endingBalanceConfirmed[M-1]  // confirmed 系列
  ```
  → 「月末残高[N] = 月末残高[N-1] + 月N収支」を forecast/confirmed の 2 系列で独立に累積する。

> **anchor 単純化規約 (MVP)**: anchor.asOf は UI で「日付」入力だが、計算の正準は **anchor.asOf が属する月を anchorMonth とし、startingBalance[anchorMonth] = anchor.balance、その月の monthDelta は date >= anchor.asOf の record のみを集計**する。anchorMonth より前の月は forecast の対象外 (forecast の from は anchorMonth 以降のみ許可)。これにより月途中 anchor でも式が一意に閉じる。

### 1.2 符号モデル (二重化禁止)

- record は `categoryId` と `signedAmount` を持つ。**`type` enum カラムは作らない**。表示上の type は `signedAmount` の符号から導出する派生値:
  ```ts
  function deriveType(signedAmount: number): "income" | "expense" {
    return signedAmount >= 0 ? "income" : "expense";
  }
  ```
- **符号強制 (proof) は書き込み時に backend が行う** (moneylog `proof_text` の翻訳):
  - `categoryId === 1` (収入カテゴリ) → `signedAmount = +Math.abs(amount)` に強制
  - `categoryId === 2` (支出カテゴリ) → `signedAmount = -Math.abs(amount)` に強制
  - それ以外の categoryId (移動/カスタム) → ユーザー入力の符号をそのまま保存 (符号フリー)
- API のリクエストは `amount` (符号付き or 絶対値どちらでも受ける) を受け、backend が上記ルールで `signedAmount` を確定して保存する。クライアントは符号正規化を行わない (予測同様、計算/正規化は backend 集約)。

### 2.2 業務テーブル (`src/db/schema.ts`)

全業務テーブルは `userId` で所有者を持ち、**全クエリは必ず `eq(table.userId, c.var.user.id)` でスコープする** (マルチテナント安全)。

```ts
import { relations, sql } from "drizzle-orm";
import { integer, sqliteTable, text, real, unique, index, primaryKey } from "drizzle-orm/sqlite-core";
import { user } from "./auth-schema";

// ---- category ----
// id=1(収入), id=2(支出) はシード。autoincrement で 3 以降がユーザー追加分。
export const category = sqliteTable("category", {
  id: integer("id").primaryKey({ autoIncrement: true }),
  userId: text("user_id").notNull().references(() => user.id, { onDelete: "cascade" }),
  name: text("name").notNull(),
  // 符号強制の挙動を id に依存させず明示フラグ化: "income" | "expense" | "free"
  signMode: text("sign_mode", { enum: ["income", "expense", "free"] }).notNull().default("free"),
  isSystem: integer("is_system", { mode: "boolean" }).notNull().default(false), // 収入/支出 = true (削除不可)
  sortOrder: integer("sort_order").notNull().default(0),
  createdAt: integer("created_at", { mode: "timestamp_ms" }).notNull().$defaultFn(() => new Date()),
}, (t) => [
  index("category_user_idx").on(t.userId),
]);

// ---- tag ----
export const tag = sqliteTable("tag", {
  id: integer("id").primaryKey({ autoIncrement: true }),
  userId: text("user_id").notNull().references(() => user.id, { onDelete: "cascade" }),
  categoryId: integer("category_id").notNull().references(() => category.id, { onDelete: "restrict" }),
  name: text("name").notNull(),
  color: text("color"),  // nullable。"#RRGGBB" or null (未指定はフロントで既定色)
  sortOrder: integer("sort_order").notNull().default(0),
  createdAt: integer("created_at", { mode: "timestamp_ms" }).notNull().$defaultFn(() => new Date()),
}, (t) => [
  index("tag_user_idx").on(t.userId),
  index("tag_category_idx").on(t.categoryId),
]);

// ---- record (単一 shape) ----
export const record = sqliteTable("record", {
  id: integer("id").primaryKey({ autoIncrement: true }),
  userId: text("user_id").notNull().references(() => user.id, { onDelete: "cascade" }),
  date: text("date").notNull(),               // "YYYY-MM-DD" (ローカル日付、TZ は Asia/Tokyo 固定 §6.1)
  yearMonth: text("year_month").notNull(),    // "YYYY-MM" (date 由来の派生。集計 index 用に冗長保持)
  signedAmount: integer("signed_amount").notNull(), // 符号付き整数(円)。収入=正/支出=負
  categoryId: integer("category_id").notNull().references(() => category.id, { onDelete: "restrict" }),
  tagId: integer("tag_id").references(() => tag.id, { onDelete: "restrict" }), // nullable
  description: text("description").notNull().default(""), // bundle 集約キー (空文字許容)
  paid: integer("paid", { mode: "boolean" }).notNull().default(true), // true=確定 / false=未確定(予定)
  // materialize 由来 record の追跡
  sourceRuleId: integer("source_rule_id").references(() => recurringRule.id, { onDelete: "set null" }),
  isManuallyEdited: integer("is_manually_edited", { mode: "boolean" }).notNull().default(false),
  createdAt: integer("created_at", { mode: "timestamp_ms" }).notNull().$defaultFn(() => new Date()),
  updatedAt: integer("updated_at", { mode: "timestamp_ms" }).notNull().$defaultFn(() => new Date()).$onUpdate(() => new Date()),
}, (t) => [
  index("record_user_ym_idx").on(t.userId, t.yearMonth),
  index("record_user_date_idx").on(t.userId, t.date),
  index("record_source_rule_idx").on(t.sourceRuleId),
  // 同一ルール × 同一月で未確定生成は 1 件のみ (二重生成防止)。
  // 手動 record (sourceRuleId=null) はこの制約に縛られない (SQLite では NULL は unique 衝突しない)。
  unique("record_rule_ym_uq").on(t.sourceRuleId, t.yearMonth),
]);

// ---- recurring rule (月次ルール) ----
export const recurringRule = sqliteTable("recurring_rule", {
  id: integer("id").primaryKey({ autoIncrement: true }),
  userId: text("user_id").notNull().references(() => user.id, { onDelete: "cascade" }),
  label: text("label").notNull(),             // ルール名 (例 "Netflix" "給料")
  dayOfMonth: integer("day_of_month").notNull(), // 1-31。月末クランプは生成時 (§5.2)
  signedAmount: integer("signed_amount").notNull(), // 符号付き整数(円)
  categoryId: integer("category_id").notNull().references(() => category.id, { onDelete: "restrict" }),
  tagId: integer("tag_id").references(() => tag.id, { onDelete: "restrict" }),
  description: text("description").notNull().default(""),
  startMonth: text("start_month").notNull(),  // "YYYY-MM" 生成開始月 (この月含む)
  endMonth: text("end_month"),                // "YYYY-MM" nullable。指定時はこの月まで(含む)。null=無期限
  active: integer("active", { mode: "boolean" }).notNull().default(true),
  createdAt: integer("created_at", { mode: "timestamp_ms" }).notNull().$defaultFn(() => new Date()),
  updatedAt: integer("updated_at", { mode: "timestamp_ms" }).notNull().$defaultFn(() => new Date()).$onUpdate(() => new Date()),
}, (t) => [
  index("recurring_rule_user_idx").on(t.userId),
]);

// ---- anchor (初期残高アンカー) : ユーザーごと 1 行 ----
export const anchor = sqliteTable("anchor", {
  userId: text("user_id").primaryKey().references(() => user.id, { onDelete: "cascade" }),
  balance: integer("balance").notNull().default(0), // 円 (整数)
  asOf: text("as_of").notNull(),                    // "YYYY-MM-DD"
  updatedAt: integer("updated_at", { mode: "timestamp_ms" }).notNull().$defaultFn(() => new Date()).$onUpdate(() => new Date()),
});

// ---- settings (materialize 窓など) : ユーザーごと 1 行 ----
export const settings = sqliteTable("settings", {
  userId: text("user_id").primaryKey().references(() => user.id, { onDelete: "cascade" }),
  materializeMonths: integer("materialize_months").notNull().default(12), // 現在月+N ヶ月先まで生成
  updatedAt: integer("updated_at", { mode: "timestamp_ms" }).notNull().$defaultFn(() => new Date()).$onUpdate(() => new Date()),
});
```

> **`signMode` を新設した理由**: 「収入=id1/支出=id2」を id 値で分岐すると、シード id がずれた瞬間に符号強制が壊れる。`signMode` 列で挙動を明示 (`income`/`expense`/`free`) し、シード時に id1→income, id2→expense を設定する。これは「type enum 二重化」ではない — `signMode` は **書き込み時の符号強制方針**であって、record の表示 type ではない (表示 type は signedAmount 符号から導出)。

### 2.3 シードデータ (新規ユーザー初回 = onboarding 時に実行)

ユーザー作成直後 (初回 API アクセス時に lazy seed、§4.7) に以下を投入:

| category.id (相対) | name | signMode | isSystem |
|---|---|---|---|
| seed1 | 収入 | income | true |
| seed2 | 支出 | expense | true |

> **注**: `category.id` はグローバル autoincrement なので、ユーザーごとに「収入カテゴリの id が 1」とは限らない。**符号強制は `signMode` 列で判定する** (id 値で判定しない)。CLAUDE.md の「id=1 収入/id=2 支出」は moneylog の単一ユーザー前提の名残であり、マルチユーザー設計では `signMode` に正規化する (この置換を Touri 承認事項とする → §承認ゲート)。

tag・anchor・settings はシードしない (anchor は未設定なら balance=0 / asOf=今日として扱う、§4.6)。

### 3.3 auth.ts (better-auth 1.6.14、Magic Link + Google OAuth)

```ts
  trustedOrigins: [process.env.BETTER_AUTH_URL!, ...(process.env.ALLOWED_ORIGINS?.split(",") ?? [])],
});
```

## 4. API / 関数シグネチャ (Hono REST)

共通規約:
- 全 `/api/*` (auth 配下除く) は `requireAuth` middleware で `c.var.user` 必須。未認証は `401 UNAUTHORIZED`。
- リクエスト body は `@hono/zod-validator` (`zValidator("json", schema)`) で検証。検証失敗は `400 VALIDATION_ERROR`。
- 全レスポンスは `200`(GET/PATCH) / `201`(POST create) / `200`(DELETE)。
- 金額は整数の円。`amount` フィールドは number (整数。zod `z.number().int()`)。
- エラーは §4.8 の **明示された有限集合** から返す (「適宜」ではない)。

### 4.1 records (`/api/records`)

```ts
// 入力 zod
const recordInput = z.object({
  date: z.string().regex(/^\d{4}-\d{2}-\d{2}$/),
  amount: z.number().int(),          // 符号付き or 絶対値。backend が proof で符号確定
  categoryId: z.number().int(),
  tagId: z.number().int().nullable().optional(),
  description: z.string().max(200).default(""),
  paid: z.boolean().default(true),
});

// 出力 record DTO
type RecordDTO = {
  id: number; date: string; yearMonth: string; signedAmount: number;
  type: "income" | "expense";        // signedAmount から導出した派生値
  categoryId: number; tagId: number | null; description: string;
  paid: boolean; sourceRuleId: number | null; isManuallyEdited: boolean;
};
```

| method | path | 入力 | 出力 | 備考 |
|---|---|---|---|---|
| GET | `/api/records?yearMonth=YYYY-MM` | query | `RecordDTO[]` | yearMonth 必須。当該月の全 record (paid 混在)、date asc |
| POST | `/api/records` | `recordInput` | `RecordDTO` (201) | proof で符号確定。yearMonth は date 由来で自動。sourceRuleId=null |
| PATCH | `/api/records/:id` | `recordInput` partial | `RecordDTO` | 所有者チェック。`sourceRuleId != null` の record を編集したら `isManuallyEdited=true` に立てる (§5.4) |
| DELETE | `/api/records/:id` | — | `{ deleted: true }` | 所有者チェック。物理削除 |

### 4.2 categories (`/api/categories`)


### 4.7.1 lazy seed / lazy materialize のフック

1. そのユーザーに category が 0 件 → §2.3 シード投入

3. rolling materialize 補充 (現在月+materializeMonths まで)

### 4.8 エラーモデル (有限集合・明示)

`AppError { status: number; code: string; message: string }`。`errorMiddleware` は **必ず status を第二引数で渡す** ([gotcha](../../../knowledge/gotcha/hono-error-middleware-apperror-status.md)):
```ts
app.onError((err, c) => {
  if (err instanceof AppError) return c.json({ error: { code: err.code, message: err.message } }, err.status);
  return c.json({ error: { code: "INTERNAL", message: "internal error" } }, 500);
});
```

**全エラーコード (これ以外は返さない)**:

| code | status | 発生条件 |
|---|---|---|
| `UNAUTHORIZED` | 401 | session 無し |
| `VALIDATION_ERROR` | 400 | zod 検証失敗 (details に zod issues) |
| `NOT_FOUND` | 404 | 対象 id が存在しない or 他人の所有 |
| `INVALID_CATEGORY` | 400 | tag/record/rule の categoryId が存在しない/他人 |
| `INVALID_TAG` | 400 | tagId が存在しない/他人/category に属さない |
| `INVALID_RANGE` | 400 | rule の endMonth < startMonth |
| `FORECAST_BEFORE_ANCHOR` | 400 | forecast from が anchorMonth より前 |
| `SYSTEM_CATEGORY_LOCKED` | 409 | isSystem category の削除 / signMode 変更 |
| `STILL_IN_USE` | 409 | §4.9 参照中の category/tag 削除 |
| `INTERNAL` | 500 | 想定外 |

### 4.9 削除ガード (STILL_IN_USE)

- `DELETE /api/categories/:id`:
  - isSystem → `409 SYSTEM_CATEGORY_LOCKED`
  - その category を参照する record / tag / recurring_rule が **1 件でもあれば** → `409 STILL_IN_USE`
  - いずれも無ければ削除
- `DELETE /api/tags/:id`:
  - その tag を参照する record / recurring_rule が 1 件でもあれば → `409 STILL_IN_USE`
  - 無ければ削除

> tag に紐づく record の `tagId` は `onDelete: restrict` なので DB レベルでも保護されるが、API は**事前 SELECT で件数を数え、明示的に STILL_IN_USE を返す** (DB 制約エラーを 500 に化けさせない)。

## 5. materialize / 集計の挙動仕様 (Reviewer のテスト根拠)

純関数は `src/services/` に置き、単体テスト可能にする。

### 5.1 materializeRule(rule, fromMonth, toMonth) — 生成

```ts
function materializeRule(ruleId: number, userId: string, fromMonth: string, toMonth: string): Promise<{ created: number }>;
```

- 対象月 = `max(rule.startMonth, fromMonth)` 〜 `min(rule.endMonth ?? toMonth, toMonth)` の各月 (両端含む)。
- 各対象月 M について生成日 = `clampDay(M, rule.dayOfMonth)` (§5.2)。
- 既に `(sourceRuleId=ruleId, yearMonth=M)` の record が存在する場合:
  - その record が `paid=true` (確定済み) → **何もしない** (保持)。
  - その record が `paid=false AND isManuallyEdited=true` → **何もしない** (手動編集を保護)。
  - その record が `paid=false AND isManuallyEdited=false` → **削除して再生成** (最新のルール内容を反映)。
- 存在しない場合 → 新規 INSERT (`paid=false`, `sourceRuleId=ruleId`, `isManuallyEdited=false`, date=clampDay, signedAmount=rule.signedAmount を proof 済み, category/tag/description=rule の値, yearMonth=M)。
- `rule.active=false` の場合: 生成は行わず、未確定&未編集の生成 record は §5.5 と同じく削除する (=非アクティブ化 = 未来の未確定予定を消す)。

### 5.2 月末クランプ (clampDay)

```ts
function clampDay(yearMonth: string, dayOfMonth: number): string; // returns "YYYY-MM-DD"
```
- その月の日数 `dim = daysInMonth(yearMonth)` (date-fns)。
- `day = Math.min(dayOfMonth, dim)`。
- 例: `clampDay("2026-02", 31) === "2026-02-28"` (閏年 2024 は `"2024-02-29"`)、`clampDay("2026-04", 31) === "2026-04-30"`、`clampDay("2026-03", 15) === "2026-03-15"`。

### 5.3 bundle 集約 (bundleRecords)

```ts
function bundleRecords(records: RecordDTO[]): BundleDTO[];
```
- `description` (空文字含む) でグルーピング。
- **2 件以上**ある description のみ Bundle 化。1 件のものは Bundle に含めない (MonthData.records 側に single として残る)。
- BundleDTO.total = Σ signedAmount、count = 件数、recordIds = 該当 record id 配列。
- 並び順: count 昇順 (moneylog 準拠)、同数は description 辞書順。
- 空文字 description の record も 2 件以上あれば 1 bundle になる (description="" の bundle)。

### 5.4 record 編集と materialize 整合

- `PATCH /api/records/:id` で `sourceRuleId != null` の record を編集した場合 → `isManuallyEdited=true` をセット。これにより以降の re-materialize で保護される。
- `paid` を false→true に変更 (確定化) した生成 record も保護される (§5.1 で paid=true は触らない)。
- 生成 record の `DELETE` は許可。削除しても次回 materialize で `(ruleId, yearMonth)` が空になり再生成される (= 「今月の予定を一旦消す」は次回補充で復活する)。**恒久的に止めたい場合は rule を編集して endMonth を縮めるか active=false にする** (UI で誘導)。

### 5.5 rule 編集 / 削除 / 非アクティブ化の semantics

- **PATCH (編集)**: rule 更新後、`fromMonth = max(rule.startMonth, currentMonth)`、`toMonth = currentMonth + materializeMonths` で §5.1 を再実行。
  - 過去月 (< currentMonth) の生成 record は **一切触らない** (確定/未確定問わず履歴として保持)。
  - 未来月の未確定&未編集生成 record のみ再生成 (新しい金額/日/category を反映)。
  - 未来月の手動編集済み (isManuallyEdited) は保持。
- **DELETE `?keepRecords=true` (既定)**: rule 行を削除。未確定&未編集の生成 record (全期間) を削除 (`removedRecords` で件数返す)。確定済み・手動編集済みの生成 record は `sourceRuleId=null` に切り離して保持 (DB の onDelete:set null)。
- **DELETE `?keepRecords=false`**: rule 行 + その rule 由来の全生成 record (確定/編集含む) を削除。
- **active=false に PATCH**: 行は残すが §5.1 で生成停止 + 未確定&未編集生成 record 削除。再度 active=true で未来分を再生成。

### 5.6 materialize トリガー (cron 不採用、lazy 採用)

- **採用**: 「書き込みトリガー (rule 作成/編集/settings 変更)」+「アクセス時 rolling 補充」の組み合わせ。**cron は採らない**。
  - 根拠: 単一ユーザー (Touri) の個人アプリ。cron インフラ (Coolify scheduled task) を別途持つ運用コストに対し、materialize は冪等で軽量 (数十 rule × 12 ヶ月 = 数百 INSERT 上限)。アクセス時に「現在月+N まで埋まっているか」を確認し不足分のみ生成する lazy 方式で十分。アクセスが無ければ古いデータでも実害なし (見るとき補充される)。
- **rolling 補充 (ensureMaterialized)**: `requireAuth` 後の共通フックで、全 active rule について `(現在月 〜 現在月+materializeMonths)` の範囲を §5.1 で冪等補充。`(sourceRuleId, yearMonth)` ユニーク制約と「未確定&未編集のみ再生成」ルールにより、毎アクセス実行しても重複生成・既存破壊は起きない。
  - 性能: 補充は「不足月のみ INSERT」。既存月は SELECT で存在確認するだけ。1 アクセスあたり active rule 数 × materializeMonths 件の存在確認 (index hit、ミリ秒未満)。

### 5.7 trend (折れ線データ) 算出

```ts
function buildTrend(yearMonth, records, startingConfirmed, startingForecast, today): TrendData;
```
- その月の 1 日〜月末の各日について 1 点。
- `forecast[day]` = `startingBalanceForecast[month] + Σ signedAmount for record where date <= day (paid 問わず)`。
- `confirmed[day]` = `today >= day ? startingBalanceConfirmed[month] + Σ signedAmount for record where date <= day AND paid=true : null` (未来日の確定線は引かない)。
- startingBalance は §1.1 の繰り越し式で前月末から得る。
- 当月でない過去月: confirmed は全日 non-null。未来月: confirmed は全日 null、forecast のみ。

### 5.8 月/年集計

- `byCategory[].all` = その月(年)・該当 category の Σ signedAmount (paid 問わず)。`paidOnly` = paid=true のみ。
- `byTag` 同様。tagId=null の record は byTag に含めない。
- `totals.incomeForecast` = Σ signedAmount where signedAmount>0 (paid 問わず)、`expenseForecast` = Σ signedAmount where signedAmount<0 (負値のまま返す)。confirmed は paid=true のみ。

## 6. 横断仕様

### 6.1 日付・タイムゾーン

- record.date / anchor.asOf / rule の月は **すべてローカル日付文字列 (`YYYY-MM-DD` / `YYYY-MM`)**。タイムゾーンは **Asia/Tokyo 固定**。「今日」は server 側で `Asia/Tokyo` の現在日を文字列で算出 (`src/lib/date.ts` の `todayJST()`)。UTC timestamp での保存はしない (日付は文字列比較で完結し TZ ずれを排除)。
- yearMonth は date の先頭 7 文字 (`date.slice(0,7)`) で導出。

### 6.2 金額

- 全金額は **整数の円** (`number`、小数なし)。zod `z.number().int()`。real/float は使わない。

### 6.3 マルチテナント

- 全業務クエリは `userId = c.var.user.id` で絞る。他人の id を指定した場合は `404 NOT_FOUND` (存在を漏らさない)。

- 多月予測: `/api/forecast?from=現在月&months=N` の series を endingBalanceConfirmed/Forecast 2 series で描画 + 各月カード。

### 7.5 定期タブ (RecurringView)

```
定期ルール                     [+ 追加]
─ 給料        毎月25日  +¥250,000  収入 ─
─ Netflix     毎月27日   -¥1,490   娯楽 ─
─ 家賃        毎月 1日  -¥65,000   住居 ─ (停止中はグレーアウト)
  行タップ → RuleSheet (編集/削除)
```

- 各 rule 行: label、`毎月X日`、signedAmount (符号色)、category/tag。active=false はグレーアウト + 「停止中」。
- [+ 追加] → RuleSheet (新規)。
- RuleSheet: label / 毎月X日 (1-31 picker) / 金額 / category 選択 / tag 選択 / description / startMonth / endMonth(任意) / active トグル。
  - 保存時 POST/PATCH → materialize → TanStack Query invalidate (`records`, `months`, `trend`, `forecast`, `recurring-rules`)。
  - 削除ボタン: keepRecords の選択 (「予定だけ消す」=keepRecords=true / 「全削除」=keepRecords=false) を confirm で出す。

### 7.6 設定タブ (SettingsView)

- **カテゴリ管理**: 一覧 (収入/支出は「システム」バッジ + 削除不可) + [+追加] → CategorySheet (name / signMode)。
- **タグ管理**: category ごとにグルーピング表示 + [+追加] → TagSheet (category / name / color picker)。color picker は固定パレット (例 8 色) + nullable。
- **初期残高アンカー**: balance + asOf 入力 → PUT /api/anchor。
- **予測期間**: materializeMonths スライダ (1-36)。
- 削除時 STILL_IN_USE は「○件で使用中のため削除できません」トースト。

## 8. テスト基盤

### 8.1 フレームワーク・配置

- **Vitest 4** (omatase-demo と整合)。
- backend テスト: `src/**/__tests__/*.test.ts` or `tests/api/*.test.ts`。`import { app } from "../../src/app"` → `app.request(path, init)`。
- frontend テスト: React Testing Library + jsdom。`tests/client/*.test.tsx`。
- 純関数テスト: `src/services/__tests__/*.test.ts` (balance/materialize/bundle/proof/clampDay を直接 import)。

### 8.2 テスト DB

- `process.env.DATABASE_URL` を test setup で `file::memory:?cache=shared` or 一時ファイルに切替。各テストで migration を流すか、`beforeEach` で truncate。
- WAL pragma はメモリ DB では無視されるが接続コードは共通でよい。

### 8.3 認証付きテスト ([gotcha: signed cookie](../../../knowledge/gotcha/better-auth-test-cookie-must-match-hono-signed-format.md))

- session 行を直接 insert し、cookie は **Hono signed cookie 形式** `better-auth.session_token=${token}.${hmac_base64}` で生成:
  ```ts
  const signature = createHmac("sha256", process.env.BETTER_AUTH_SECRET!).update(token).digest("base64");
  const cookie = `better-auth.session_token=${token}.${signature}`;
  ```
- helper は `tests/helpers/auth.ts` に集約 (`createUserAndCookie()` → `{ userId, cookie }`)。各 test file から直接 cookie を組まない。
- better-auth handler が `app.on(["POST","GET"], "/api/auth/**", ...)` で mount されているか、401 が消えない場合は `/api/auth/get-session` の status で切り分け。

### 8.4 主要テストパターン (Reviewer 指針)

- **純関数 (最優先・確実 GREEN)**: clampDay (端日)、proof (符号強制 3 分岐)、bundleRecords (2件未満は束ねない/count昇順/空description)、buildTrend (今日以降 confirmed=null)、繰り越し累積 (§1.1)。
- **materialize**: 新規生成 / 既存 paid=true 保持 / isManuallyEdited 保持 / 未確定未編集の再生成 / `(ruleId,yearMonth)` ユニーク (二重生成しない) / active=false で削除 / endMonth クランプ。
- **API integration**: 各 CRUD の 201/200/404、符号強制 (POST income→正)、STILL_IN_USE (使用中 category/tag 削除で 409)、SYSTEM_CATEGORY_LOCKED、FORECAST_BEFORE_ANCHOR、VALIDATION_ERROR、マルチテナント (他人 id で 404)。**status code を必ず assert** ([gotcha](../../../knowledge/gotcha/hono-error-middleware-apperror-status.md))。
- **描画**: RecordRow (paid=false で unpaid-mark)、BundleRow (text="desc (count)")、SummaryCard (currentBalance=null 非表示)、LineChart (null 点で polyline 分割)、Sheet (3 経路 close)。
- frontend で getBoundingClientRect=0 等の jsdom 制限は LineChart の viewBox 計算が DOM 寸法非依存 (props の数値のみ) なので回避済み。

## 9. 不採用案 (再検討ループ防止)

| 案 | 却下理由 |
|---|---|
| **RRULE (RFC5545)** | 要件は FREQ=MONTHLY 単一。週次/BYDAY/外部カレンダー互換不要。rrule npm のパーサ依存はオーバーキル。`dayOfMonth` 整数 1 列で表現でき、端日は clampDay 1 関数で済む |
| **on-the-fly 展開** (予測時にルールを毎回展開) | moneylog の「record を月に積む」一元モデルから乖離。予測パスが定期/手動を区別する分岐を抱える。materialize なら予測は record 集計のみで定期を意識しない (uniform shape)。「今月の給料だけ金額変更」も生成済み record の直接編集で済む |
| **cron で materialize** | 単一ユーザー個人アプリで cron インフラ (Coolify scheduled task) を持つ運用コストに見合わない。lazy (書き込み + アクセス補充) で冪等・軽量に足りる。アクセスしない期間に古くても実害なし |
| **クレカ請求サイクル** (利用→翌月引き落とし) | CLAUDE.md でやらない確定。クレカは「毎月固定日の固定額支出」= 通常の月次ルールで表現 |
| **moneylog の単一 POST + control ディスクリミネータ API コピー** | CLAUDE.md でコピー禁止確定。Hono REST で resource ごとにルートを切る方が型安全・テスト容易・TanStack Query invalidate と素直に対応 |
| **重量級グラフライブラリ (Chart.js / Recharts / ECharts)** | 2 系列の単純折れ線 + 縦線 + 月カードに数百 KB は過大。moneylog も自前 graph 実装。自前 `<LineChart>` (SVG polyline) で props 駆動 → 描画テストも DOM 寸法非依存で書ける。phase2 で tooltip 要求が出たら再考 |
| **type enum カラムを record に持たせる** | 符号モデルと二重化し不整合源になる (符号は正だが type=expense 等)。type は `signedAmount` 符号から導出する派生値に統一。符号強制は category の `signMode` 列で行う (これは type ではなく書き込み方針) |
| **id=1収入/id=2支出 を id 値で判定** | マルチユーザーで autoincrement id がユーザーごとにずれると符号強制が壊れる。`category.signMode` 列で挙動を明示し id 非依存にする |
| **anchor を record として持つ** (moneylog 流の起点 record) | record テーブルに「残高調整」という別概念を混ぜると bundle/集計/符号導出が濁る。anchor 専用 1 行テーブルで起点を明示する方が累積式 (§1.1) が読みやすい |

## 10. Touri が承認すべき判断点

1. **`signMode` 列の新設**: CLAUDE.md は「id=1収入/id=2支出」だが、マルチユーザー安全のため id 値判定をやめ `category.signMode` (income/expense/free) 列で符号強制を明示する正規化。型/挙動は同じ (収入カテゴリは符号+強制、支出は−強制、他は free)。
2. **materialize トリガー = lazy (cron 不採用)**: 書き込み時 + アクセス時 rolling 補充。cron インフラを持たない。
3. **TZ = Asia/Tokyo 固定 + 日付は文字列保持** (UTC timestamp で持たない)。
4. **グラフ自前 SVG `<LineChart>`** (外部ライブラリ不使用)。
5. **anchor 専用テーブル** (record として持たない) + anchor.asOf は月初推奨 (月途中は §1.1 単純化規約で一意化)。
6. **rule 削除の既定 = keepRecords=true** (確定/編集済みは保持、未確定未編集の予定のみ削除)。
